CTMediator 组件化实践

发表于 2018-10-30 | 分类于 iOS开发 |

CTMediator

关于 CTMediator 其实接触已经有一段时间了，只是前段时间一直在忙着换工作，准备面试，还有出去旅游，~~ 所以一直没有写这篇文章。
写这个之前，我已经将上家公司的项目 使用 CTMediator 组件化完成了，也算是对之前的东家有个好的交代，希望发展的越来越好。

其实作者本身关于 CTMediator 的文章已经很详细了，而且 CTMediator 源码也并不复杂，可以说实现方法 以及 源码 都比较通俗易懂。

这里就谈谈自己的收获吧，

  • 关于组件 与 基本库

在分模块的时候，要搞清楚这个模块是时候作为组件，还是适合作为基本库来独立出来。

  • 关于去 model 化

老实说，我一直对于所谓的去 model 化的网络层方案之类的是不屑一顾的，因为 model 确实很方便啊，如果去 model 全部都用 通用数据对象（NSDictionary）来表示的话，需要另一套行之有效的方案，而且其他开发人员的学习成本也是比较高的。但是 经过这次组件化实践之后，发现，去 model 化还是必要的，因为在整个过程中，多次出现多个 组件 依赖同一个 model 的情形，最后无奈只能讲依赖同一个 model 的逻辑再度 组件化，其实这样做，我觉得是没有必要的，所以网络层 是应该需要做去model化的处理的。只是需要一个高效，学习成本低的方案。而 组件间调用 去 model 化，确实是显而易见的必须的，不然根本不解耦。

  • 关于网络层

上面提到，因为项目的网络层是 model 化的，所以导致无法解耦，这是其一，其二 就是，原先的网络层设计是采用的 协议的方式，因为项目是采用的 MVVM 的设计模式，所以网络层 基本是只在 VM 中调用的，最后还是无法将网络层 单独的当做一个组件来抽取，只能当做基本库，存在于项目中。

  • 关于持久层

项目采用的是 Realm 数据库，持久层也是 非去 Model 化的，虽然能将持久层单独的组件化，但是不够灵活，而且对外提供的 target 业务较多。

  • 关于通用配置的 常量 && 一些全局属性获取

先说一说之前的状况，因为 组里有个实习的小朋友， 他有个习惯，每次写代码，刷刷刷，一顿操作，然后 觉得有些 常量 或者 全局属性需要获取的时候，去到项目的 配置文件，一顿添加，最后你发现，这些添加其实都是非必要的，哈哈哈哈，现在就没这个烦恼了，组件化之后，要修改这些配置文件，就没那么随意了，~~
